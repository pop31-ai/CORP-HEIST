## 0. ГЛАВНЫЙ ЛИМИТ — постоянные коннекты (запомнить!)

> **≈ 30 одновременных постоянных соединений — потолок.**
> Каждый висящий коннект держит память; сверх лимита память переполняется →
> Linux уходит в OOM и **машина падает**. Это причина №1 всей парадигмы транспорта.

Отсюда следуют все ключевые решения:

| Решение | Как защищает от лимита коннектов |
|---|---|
| **Один порт `9000`** | не провоцирует держать по коннекту на порт (в старой 30-портовой модели соблазн — 30 висящих соединений) |
| **Pulse+Poll, без WebSocket** | коннект живёт доли секунды: `GET/POST → ответ → закрыт`. В любой момент открыто лишь несколько, а не по одному на каждого игрока |
| **Состояние в общей памяти, не в соединении** | клиент stateless между запросами → соединение можно закрывать сразу |
| **Файловый JSON, короткие ответы** | быстрый ответ = коннект быстро освобождается |

### Почему poll укладывается в лимит

Постоянных коннектов держится столько, сколько запросов **обрабатывается прямо
сейчас**, а не сколько всего игроков. При коротком ответе (чтение из RAM, единицы мс)
даже тысячи игроков дают в каждый момент лишь горстку одновременных соединений:

```
одновременные_коннекты ≈ RPS × длительность_ответа

Пример: 2000 запросов/с × 0.003 с (ответ из RAM) ≈ 6 одновременных коннектов
        → с большим запасом под потолком ~30.
```

Наоборот, WebSocket дал бы **1 постоянный коннект на каждого онлайн-игрока** —
30 игроков и машина на грани. Именно этого мы избегаем.

> ⚠️ **Противопоказано:** multi-worker / SO_REUSEPORT / много портов ради
> «разгрузки» — они плодят висящие коннекты и гарантированно уронят машину.
> Масштабирование — только через *короткие* запросы и (если совсем нужно)
> отдельные VM с общим Redis, а НЕ через рост числа постоянных соединений.

|---|---|

| Disk | ~32 GB |
| CPU | 2 vCPU |

| Внешних портов | 1 (`:9000`) |

> Точный объём RAM боевой VM не подтверждён — расчёты ниже даны оценочно.
> Лимит ~30 коннектов — жёсткий инвариант **независимо** от RAM.

## 2. Бюджет памяти

| Ресурс | Оценка |
|---|---|
| Python-процесс (aiohttp + код) | ~45 MB |
| chars[500] (~2 KB/шт) | ~1 MB |
| stocks[20] | ~10 KB |
| Открытые коннекты (aiohttp) | зависит от их числа — **держим ≤ ~30** |
| HTML/JS ассеты | ~50 KB |
| **Итого (idle)** | **~52 MB** |

Масштаб персонажей: до 10K в том же процессе (~20 MB) — всё ещё < 100 MB.
Диск: код+данные ~50 MB из 32 GB — запас 99.8%.

## 4. Трафик при нагрузке (10K игроков, оценка)

| Метрика | Наивно | Binary |
|---|---|---|
| Client→Server /с | 46 MB/s | 4.3 MB/s |
| Server→Client /с | 46 MB/s | 4.3 MB/s |
| P2P /с | 20 MB/s | 0 B |
| **Итого /с** | **112 MB/s** | **8.6 MB/s** |
| Экономия | — | 92.3% |

P2P = 0: «инстансы» живут в одном процессе через `SharedState`, между ними
0 байт и 0 мс (ссылка на dict в общей памяти).

## 5. Транспорт: Pulse+Poll vs WebSocket (память — решающее)

| Модель | Idle /с | Active /с | RAM (10K) | Постоянных коннектов |
|---|---|---|---|---|
| WebSocket | 0 B | 4.3 MB/s | ~80 MB | **~10K (❌ мгновенный OOM)** |
| Pulse+Poll | 0 B | 3.1 MB/s | ~12 MB | **горстка (✅ ≤ ~30)** |

Именно последний столбец — причина выбора. Экономия RAM ~85%, но главное —
**число постоянных коннектов остаётся низким**.

Размеры pulse-эндпоинтов (binary-в-JSON):

| Endpoint | Req | Resp |
|---|---|---|
| /pulse/trade | 40 B | 30 B |
| /pulse/gacha | 12 B | 20 B |
| /pulse/loot/sell | 24 B | 25 B |
| /pulse/donate | 12 B | 30 B |
| /pulse/buy-gold | 20 B | 30 B |

## 6. Выводы

1. **Лимит ~30 постоянных коннектов** — жёсткая граница; всё спроектировано вокруг неё.
2. Много портов **не дают** разгрузки (один event loop, один GIL) — они лишь
   провоцируют лишние висящие коннекты. Один порт правильнее.
3. Pulse+Poll держит коннекты короткими → тысячи игроков умещаются под потолком.
4. Binary-протокол снимает ~90% трафика; RAM в idle ~52 MB — free-tier ок (0 ₽).
5. Настоящее масштабирование, если понадобится: **короткие запросы + отдельные
   VM с общим Redis**, НЕ рост числа постоянных соединений и НЕ multi-port.
